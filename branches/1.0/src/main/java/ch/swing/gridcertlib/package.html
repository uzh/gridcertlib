Package org.swing.gridcertlib provides services to create a SLCS/X.509 certificate and a Grid proxy (optionally VOMS-enabled), given the SAML2 assertion resulting from a Shibboleth2 authentication.

The library was written with a specific use-case in mind: use within a portal or web application. The programming interface has thus been kept generic, in order to adapt to the various programming paradigms used by different host products. The only hard requirement is that certificates and proxies can be stored on the filesystem.

Package {@link ch.swing.gridcertlib.servlet} provides example servlets (with fully-commented code) that show how the library can be used.

Deployment and configuration caveats are discussed in the {@link ch.swing.gridcertlib.servlet} package overview.

Programming interface

The library package has two public entry points:

{@link ch.swing.gridcertlib.SLCSFactory}
Façade to request SLCS certificates and store them to a chosen location on the filesystem.
{@link ch.swing.gridcertlib.GridProxyFactory}
Façade to create Globus Toolkit proxy certificates (optionally requiring VOMS extensions) and store them to a temporary location on the filesystem.
A single instance of each of these classes can generate multiple SLCS certificates or proxies (possibly for different users) via repeated invocation of the {@link ch.swing.gridcertlib.SLCSFactory#newSLCS} / {@link ch.swing.gridcertlib.GridProxyFactory#newProxy} methods.

Since the parameters used to configure the factory objects are portal-wide globals and unlikely to change much over time, the configuration can be passed in through a {@link java.util.Properties} object, which can conveniently be loaded from a file (see an example in the {@link ch.swing.gridcertlib.servlet.SlcsInit} servlet).
